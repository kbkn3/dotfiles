# General instructions

- 日本語で応答します
- 残念だが多くのコメントには見ればわかることが書いていることが多い、数行読めばわかるコメントに価値はない。しかし、あえてやらなかったことはコメントからしかわからない。そういう情報がコメントとしてほしい。なぜその処理が必要なのか
- 関数名や変数名で `cfg` や `env` といった省略は使わず、`configuration` や `environment` のように省略せず英単語として正しく記述する
- File System Hierarchy StandardやSemantic Web, RFCなど標準に徹底的に従う
- 軽微な実装で済む場合は外部ライブラリの利用を避けて自前で実装する
- 余計な空白をファイルに追加してはいけない、インデントも守りましょう
- 変更は最小限にとどめる
- リポジトリの癖があると思う。既存のパターンに従うようにする
- 徹底的なテスト

## コーディング

コードタスクを支援する際のガイドラインは以下の通りです：

1. **コード分析**
   - 提案を行う前にコードを徹底的に分析
   - プロジェクトの構造や慣習を理解
   - 潜在的なエッジケースやバグを識別

2. **解決策の設計**
   - 清潔で保守しやすい解決策を優先
   - パフォーマンスと可読性のバランスを取る
   - 既存のコードベースのスタイルとパターンに従う

3. **実装**
   - 簡潔でよく構造化されたコードを書く
   - 必要に応じて依存性を最小限に抑える
   - 適切なエラーハンドリングを含む
   - 適用できる場合は型の安全性を確保
   - 言語固有の慣習やガイドラインを厳守
   - 言語の慣用パターンやベストプラクティスに従う

4. **テストと検証**
   - 適切なリンターや型チェックを実行
   - 変更が期待通りに動作することを確認
   - テストではエッジケースを考慮
   - 明示的に求められない限りテストは作成しない

5. **ドキュメンテーション**
   - コメントは最小限かつ情報を伝える内容に
   - 必要に応じてドキュメントを更新

6. **言語標準**
   - 各言語の公式スタイルガイドに従う（PythonならPEP 8、GoならGoogleスタイルガイドなど）
   - ライブラリ・フレームワークについては Context7 MCP を利用して調べる
   - 言語固有のイディオムを適切に使用
   - 命名、フォーマット、組織に関する言語の慣習を尊重

7. コメント
    - コメントを最小にする
    - コードの「why」を説明する意味のあるコメントにする

## Example Workflow

以下の手順に従って開発してください。

**`TodoWrite` の際に注意深く考慮すること**

1. 関連性の高い既存のコードベースを探し、見つかったコードスタイルに準拠するようにする
2. 差分を最小限に保ちながら実装する
3. 未使用の関数や変数を削除する
4. 処理内容から自明なコードコメントを削除する
5. テストやコンパイルが通ることを確認する
6. これまでの指示を遵守していることを確認する
7. 最終的な成果物をパフォーマンス・セキュリティといった観点で否定的にレビューし、今後の改善点を示す
