# pr-create

GitHub Pull Request を作成するためのスキル。リポジトリに PR テンプレートが存在する場合は必ずそれに従い、存在しない場合は Claude Code のデフォルト形式を使用する。

## なぜテンプレートに従うのか

PR テンプレートはチームのレビュー規約・記入すべき情報の合意そのもの。勝手に独自フォーマットで書くと、レビュワーが必要な情報を得られず、CI/チェックリストが機能しない。テンプレートがあるときはそれが優先される。

## 実行手順

### 1. ブランチ状態を把握

以下を並列で実行してブランチの状態を確認する。

- `git status`（`-uall` は使わない）
- `git diff`（ステージング済み + 未ステージを両方）
- `git log <base>..HEAD`（base はデフォルトで `main` または `develop`、リポジトリに応じて判定）
- `git diff <base>...HEAD`（分岐後の全差分）
- リモート追跡ブランチの状態（push が必要か）

### 2. PR テンプレートを探す

リポジトリルートで `.github/pull_request_template.md` を Read する。

- **存在する場合**: そのテンプレートの構造（見出し・チェックリスト・HTMLコメント含む）を**一字一句崩さず**維持したまま、埋められる項目だけ埋める。
- **存在しない場合**: 後述の「デフォルト形式」を使う。

`.github/PULL_REQUEST_TEMPLATE/` ディレクトリ（複数テンプレート）や他パスは、このスキルでは対応しない。

### 3. テンプレートを埋める（テンプレートがある場合）

**埋め方の原則:**

1. コミット履歴・差分から客観的に読み取れる情報のみ埋める。
2. 埋められない項目（「関連Issue」「スクリーンショット」「動作確認手順」など、コード差分から推測できないもの）は、**テンプレートのプレースホルダや空欄のまま残す**。勝手に "N/A" や "なし" を書き込まない。
3. HTML コメント（`<!-- ... -->`）はそのまま残す。レビュワーへの注記であり、書き換えてはいけない。
4. チェックボックス（`- [ ]`）は、差分から明らかに満たしていると判定できない限り未チェックのまま。
5. 既存の見出し・順序・箇条書き記号を変更しない。

**埋めてよい例:**
- 「## 概要」「## Summary」→ コミット履歴と差分から要約を生成
- 「## 変更点」「## Changes」→ 主要な変更ファイル・機能を箇条書きで記述
- 「## Motivation」「## なぜ」→ コミットメッセージの本文やコンテキストから抽出

**埋めない例:**
- 「関連Issue: #___」→ ユーザーが指定しない限り空欄
- 「スクリーンショット」→ UI変更でも自動生成できないので空欄
- 「動作確認手順」→ テンプレートがプレースホルダを持つならそのまま残す
- 「レビュワー指定」→ 空欄

### 4. デフォルト形式（テンプレートがない場合のみ）

Claude Code 既定の形式を使う。

```markdown
## Summary
<1-3 bullet points>

## Test plan
[Bulleted markdown checklist of TODOs for testing the pull request...]
```

### 5. タイトルを決める

- Conventional Commits を採用しているリポジトリ（`CLAUDE.md` / `AGENTS.md` / `.commitlintrc` 等で確認、または既存の最近のコミット/PRが従っている）では、`<type>(scope): <description>` 形式。
- そうでないリポジトリでは、変更を端的に表す短い文（70文字以内）を英語または日本語で書く。既存PRの書き方に合わせる。
- タイトルは簡潔に。詳細は body に書く。

### 6. ブランチを push（必要なら）

- 現ブランチがリモートに無い場合は `git push -u origin <branch>`。
- 追跡済みだが未push のコミットがある場合は `git push`。

### 7. `gh pr create` を実行

body は HEREDOC で渡す（フォーマット崩れ防止）。

```bash
gh pr create --title "<title>" --body "$(cat <<'EOF'
<生成した body>
EOF
)"
```

base ブランチがデフォルト以外のときのみ `--base <branch>` を付ける。

### 8. 結果をユーザーに返す

`gh pr create` が返した PR URL をそのままユーザーに伝える。

## やってはいけないこと

- **テンプレートを勝手に作り変えない**: 見出し追加・削除、順序変更、HTMLコメント削除は禁止。
- **埋められない項目を捏造しない**: 関連 Issue 番号・動作確認手順・スクリーンショット等、事実として確認できないものを埋めない。プレースホルダのまま残す方が、レビュワーに「まだ未記入」と伝わって親切。
- **チェックボックスを勝手にチェックしない**: 本当に満たしていると差分から判定できる項目以外は未チェックのまま。
- **`git push --force` / `--force-with-lease`**: ユーザーが明示的に要求しない限り使わない。
- **PR作成前にコミットする**: スキル発動時点で未コミットの変更があっても、勝手に `git commit` しない。ユーザーに確認する。

## 判断に迷ったら

- テンプレートの意図が読み取れない項目は空欄のまま残し、ユーザーに「この項目は自動では埋めませんでした」と一言伝える。
- base ブランチの判定が曖昧なとき（main/develop/master が混在するなど）は、`gh repo view --json defaultBranchRef` で確認してから使う。
- 複数コミットがある PR では、全コミットを俯瞰して body を書く。最新コミットだけを見て要約しない。

## 運用上の注意（プロジェクトへの組み込み）

単純な PR 作成依頼（「PR 作って」など 1 ステップで済む依頼）では、Claude が自前の判断で `gh pr create` を直接実行してしまい、このスキルを参照しないことがある。**スキルが必ず発火するようにしたいリポジトリ**では、リポジトリの `CLAUDE.md` もしくは `AGENTS.md` に次の一文を入れること:

```markdown
PR / Pull Request を新規作成するときは、必ず `pr-create` スキルに従うこと（テンプレート遵守のため）。
```

これによりリポジトリのコンテキストでスキル使用が明示され、単純な依頼でもテンプレート規約が守られる。
